# 폴더별 역할 정리

이 문서는 식재료 기반 요리 추천, 유통기한 관리, 알림, 식재료 공유/교환 커뮤니티 웹 사이트의 초기 폴더 구조와 각 폴더의 역할을 정리한 문서입니다.

프로젝트는 전통적인 CRUD 흐름을 기준으로 구성합니다.

```text
View -> Controller -> Service -> Mapper -> DB
```

## 전체 구조 예시

```text
project-root/
├── backend/
│   ├── controllers/
│   ├── services/
│   ├── mappers/
│   ├── models/
│   └── config/
├── frontend/
│   ├── pages/
│   ├── css/
│   └── js/
├── database/
└── docs/
```

## `backend/`

C++와 Drogon으로 작성되는 백엔드 서버 코드가 들어가는 최상위 폴더입니다.

사용자의 요청을 받고, 비즈니스 로직을 처리하며, MySQL 데이터베이스와 통신하는 전체 서버 로직을 담당합니다.

## `backend/controllers/`

클라이언트의 HTTP 요청을 가장 먼저 받는 계층입니다.

예를 들어 다음과 같은 요청을 처리합니다.

- 식재료 목록 조회
- 식재료 등록, 수정, 삭제
- 보유 식재료 기반 레시피 추천 요청
- 유통기한 임박 식재료 조회
- 커뮤니티 게시글 작성 및 조회
- 식재료 교환 요청

Controller는 요청과 응답을 담당하고, 실제 핵심 로직은 Service에 위임합니다.

## `backend/services/`

서비스의 핵심 비즈니스 로직을 담당하는 계층입니다.

예를 들어 다음과 같은 로직이 들어갑니다.

- 사용자가 가진 식재료와 레시피 재료를 비교해 추천 레시피 계산
- 유통기한이 임박한 식재료 판단
- 알림 생성 조건 판단
- 커뮤니티 게시글 작성 권한 확인
- 식재료 교환 요청 상태 변경

Service는 Controller와 Mapper 사이에서 데이터를 가공하고 정책을 적용합니다.

## `backend/mappers/`

MySQL 데이터베이스와 직접 통신하는 계층입니다.

SQL 쿼리를 실행하거나 Drogon의 DB Client를 사용해 데이터를 조회, 저장, 수정, 삭제합니다.

예를 들어 다음과 같은 작업을 담당합니다.

- 사용자 정보 조회
- 식재료 데이터 저장
- 레시피 목록 조회
- 알림 데이터 저장
- 커뮤니티 게시글 조회
- 교환 요청 상태 업데이트

Mapper는 DB 접근만 담당하고, 비즈니스 판단은 Service에서 처리합니다.

## `backend/models/`

백엔드에서 사용하는 데이터 구조를 정의하는 폴더입니다.

예를 들어 다음과 같은 모델을 둘 수 있습니다.

- `User`
- `Ingredient`
- `Recipe`
- `Notification`
- `CommunityPost`
- `ExchangeRequest`

DB 테이블과 직접 연결되는 구조체나, API 응답에 사용하는 데이터 객체를 정의합니다.

## `backend/config/`

서버 실행에 필요한 설정 파일을 관리합니다.

예를 들어 다음과 같은 설정이 들어갈 수 있습니다.

- Drogon 서버 포트
- MySQL 연결 정보
- 로그 설정
- CORS 설정
- 실행 환경 설정

민감한 정보는 코드에 직접 작성하지 않고, 별도의 환경 변수나 설정 파일로 분리하는 것이 좋습니다.

처음에는 꼭 필요할 때만 추가하고, 너무 많은 기능을 미리 넣지 않는 것이 좋습니다.

## `frontend/`

HTML, CSS, JavaScript로 작성되는 프론트엔드 코드가 들어가는 최상위 폴더입니다.

사용자가 직접 보는 화면과 브라우저에서 실행되는 로직을 담당합니다.

## `frontend/pages/`

각 화면의 HTML 파일을 관리합니다.

예를 들어 다음과 같은 페이지를 둘 수 있습니다.

- 메인 페이지
- 로그인 페이지
- 회원가입 페이지
- 내 식재료 관리 페이지
- 레시피 추천 페이지
- 유통기한 관리 페이지
- 알림 페이지
- 커뮤니티 페이지
- 식재료 교환 페이지

처음에는 기능별로 단순한 HTML 파일을 나누는 방식으로 시작합니다.

## `frontend/css/`

화면 스타일을 담당하는 CSS 파일을 관리합니다.

예를 들어 다음과 같이 나눌 수 있습니다.

- 공통 스타일
- 레이아웃 스타일
- 페이지별 스타일
- 버튼, 입력창, 카드 같은 UI 요소 스타일

초기에는 `style.css` 하나로 시작하고, 파일이 커지면 기능별로 분리해도 됩니다.

## `frontend/js/`

브라우저에서 실행되는 JavaScript 파일을 관리합니다.

예를 들어 다음과 같은 역할을 담당합니다.

- 백엔드 API 호출
- 화면 데이터 렌더링
- 입력값 검증
- 버튼 클릭 이벤트 처리
- 알림 목록 표시
- 추천 레시피 결과 표시

처음에는 페이지별 JS 파일을 두는 방식이 가장 단순합니다.

## `database/`

MySQL 데이터베이스와 관련된 파일을 관리합니다.

예를 들어 다음과 같은 파일을 둘 수 있습니다.

- 테이블 생성 SQL
- 초기 테스트 데이터 SQL
- DB 구조 변경 기록

초기에는 `schema.sql` 파일에 기본 테이블 생성문을 정리하는 방식이 적합합니다.

## `docs/`

프로젝트 문서들을 관리하는 폴더입니다.

예를 들어 다음과 같은 문서를 둘 수 있습니다.

- 프로젝트 개요
- API 명세
- DB 설계
- 개발 규칙
- 실행 방법

초기에는 문서를 너무 많이 나누지 않고, 필요한 문서부터 하나씩 추가하는 것이 좋습니다.

## 요청 처리 흐름 예시

식재료 등록 기능을 예로 들면 다음 흐름으로 동작합니다.

```text
사용자 화면
-> frontend/js에서 API 요청
-> backend/controllers에서 요청 수신
-> backend/services에서 등록 로직 처리
-> backend/mappers에서 MySQL에 저장
-> 결과를 다시 프론트엔드에 응답
```

## 초기 개발 방향

처음부터 복잡한 구조를 만들기보다 다음 순서로 확장하는 것이 좋습니다.

1. 기본 폴더 구조 생성
2. MySQL 테이블 초안 작성
3. 식재료 CRUD 구현
4. 유통기한 관리 기능 추가
5. 레시피 추천 기능 추가
6. 알림 기능 추가
7. 커뮤니티 및 교환 기능 추가

초기 목표는 모든 기능을 완벽하게 만드는 것이 아니라, 각 계층이 어떤 역할을 하는지 명확히 나누고 단순한 CRUD 흐름을 먼저 완성하는 것입니다.
